Why is Software Development difficult?

  • The problem is usually ambiguous.
  • The requirements are usually unclear and changing when they become clearer.
  • The problem domain (called application domain) is complex, and so is the solution domain.

Why is Software Development difficult?

  • The development process is difficult to manage.
  • Software offers extreme flexibility.
  • Software is a discrete system
    • Continuous systems have no hidden surprises.
    • Discrete systems can have hidden surprises.

What is Software Engineering (SE)

  • is a modeling activity.
  • is a problem-solving activity.
  • is a knowledge acquisition activity.
  • is a rationale-driven activity.

Modeling

  • model is an abstract representation of a system that enables us to answer questions about the system.
  • Models are useful when dealing with systems that are too large, too small, too complicated, or too expensive to experience firsthand.
  • Models also allow us to visualize and understand systems that either no longer exist or that are only claimed to exist.

Modeling

  • Object-oriented methods application domain + solution domain.
  • Example: a train traffic control system includes train objects representing the trains it monitors is part of the application domain.
  • The set of lines used to depict a train are part of the solution domain.
  • The idea of object-oriented methods is that the solution domain model is a transformation of the application domain model.

Software Engineering Concepts

Software Engineering Concepts

  • Participants persons involved in the project.

  • System collection of interconnected parts.
  • Model any abstraction of the system.
  • Work Product an artifact produced during the development.
  • Deliverable work product that must be delivered to a client.

Software Engineering Concepts - Example

  • TicketDistributor is a machine that distributes tickets for trains.
  • Travelers have the option of selecting a ticket for a single trip or for multiple trips, or selecting a time card for a day or a week.
  • The TicketDistributor computes the price of the requested ticket based on the area in which the trip will take place.
  • The TicketDistributor must be able to handle several exceptions, such as travelers who do not complete the transaction, travelers who attempt to pay with large bills, and resource outages, such as running out of tickets, change, or power.

Examples of roles in software engineering for the TicketDistributor project

  • Client responsible for providing high-level requirements on the system.
  • User responsible for providing domain knowledge of current user tasks.
  • Manager responsible for the work organization.
  • Human Factors Specialist responsible for the usability of the system.
  • Developer responsible for the construction of the system.
  • Technical Writer responsible for documentation delivered to the client.

Examples of work products for the TicketDistributor project

  • Specification describes the system from the users point of view.
  • Operation manual used by the staff of the train company responsible for installing and configuring the TicketDistributor.
  • Status report describes at a given time the tasks that have been completed and the tasks that are still in progress.
  • Test manual tracks the known defects in the prototype TicketDistributor and their state of repair.

Examples of activities, tasks, and resources for the TicketDistributor project

  • Activity Requirements elicitation.
  • Task Develop Out of Change test case for TicketDistributor.
  • Task Review Access Online Help use case for usability Tariff Database.
  • Resource Tariff Database.
